# drv_laser 抽象统一接口与复杂度评估（2026-03-31）

## 1) 统一接口在哪一层

结论：统一接口不是一层，而是两层协同。

- L1（驱动统一接口层）
  - 位置：`drivers/drv_interface.h`
  - 形式：`drv_interface_t`（`init/enable/disable/get_state/force_off/idle_tick`）
  - `drv_laser` 的挂接点：`drivers/drv_laser.c` 里的 `g_laser_interface` + `laser_get_interface()`

- L2（会话统一入口层）
  - 位置：`hw_system/hw_system.c`, `hw_system/hw_system.h`
  - 形式：`hw_system_request(START/CONFIG/STOP)`
  - 含义：业务层不直接拼驱动调用顺序，统一通过会话入口控制硬件链路。

一句话：L1 统一“单驱动契约”，L2 统一“跨驱动会话”。

## 2) drv_laser 复杂度评估

- 文件：`drivers/drv_laser.c`
- 行数：约 `521`
- 复杂度判定：`中高`

主要复杂来源（同一文件承载了多职责）：

1. 驱动接口适配职责
- 包含 `drv_interface_t` 适配函数（`laser_drv_*`）与导出接口。

2. 硬件后端分流职责
- 同时支持 GPIO 方式和 DAC 方式。
- 还包含 profile 分支（单发/双发差异）。

3. 功率策略职责
- 普通功率、双发功率、光路相关功率分配都在本文件。

4. 配置持久化职责
- 直接含 `KV` 读取/写入逻辑（功率配置）。

5. 运行态附加职责
- 闪烁、快速开关、日志输出等运行行为也在本文件。

评估结论：
- 代码本身可用，但“职责密度偏高”，阅读成本高。
- 维护者容易在“硬件细节、会话语义、策略逻辑”之间来回跳转。

## 3) 当前注释状态判断

- `drv_laser.c` 顶部已有模块注释和部分中英注释。
- 但对“对外统一契约 + 内部分层边界”说明不足。
- 现状更像“有局部注释”，不是“一眼可维护的抽象文档化”。

## 4) 建议的收敛方式（不改行为）

先做注释收敛，再做文件拆分（可选）：

1. 注释收敛（必须）
- 在 `drv_laser.c` 增加“文件内分层图”注释：
  - 接口适配层
  - 硬件后端层（GPIO/DAC）
  - 策略层（单发/双发功率）
  - 配置层（KV）
- 明确哪些 API 允许业务层调用，哪些仅供 `hw_system` 调用。

2. 结构收敛（建议）
- 拆为：
  - `drv_laser_iface.c`（接口与状态机）
  - `drv_laser_hw.c`（GPIO/DAC 后端）
  - `drv_laser_policy.c`（单发/双发策略）
  - `drv_laser_cfg.c`（KV 配置）
- 头文件保留统一 API，不改调用方。

## 5) 对齐要求映射（你关心的点）

- “统一接口在哪层”：
  - 单驱动接口：L1
  - 统一会话入口：L2

- “为什么看着一层套一层”：
  - 不是坏事本身，问题在于同文件职责太多，层次注释不够明确。

- “怎么落地”：
  - 先补注释把层次写清，再决定是否拆文件；先不改行为最稳。
