原理切换先后顺序（必须按这个改）

  1. 先定身份：`BOARD_TYPE`（`board/board.h`）与 `PROFILE_TYPE`（`profile.h`）一致。
  2. 再定原理：`OPTICAL_MODE`（`config.h`）确保与目标原理一致。
  3. 再校引脚：对应 `board_gd303_*.h` 的采样脚、触发脚、功率脚、LD_SELECT。
  4. 再看能力宏：`USE_DUAL_ADC_SYNC`、`USE_DAC_POWER`、`USE_LD_SELECT` 是否符合目标原理。
  5. 再核驱动：`drv_adc/drv_laser/drv_optical_switch` 是否与能力宏一致执行。
  6. 最后核采集语义：`measure_session/measure_core` 的 `outer/inner` 解包和 path 语义。

统一流程与替换边界

  • 流程基线保持 unified：`HW_REQ_START -> HW_REQ_CONFIG -> HW_REQ_STOP`。
  • 采集替换优先落点：`drv_adc`、`measure_session`、`measure_core`，不先动 S3/S4 融合。
  • 上层命令链用于回归闭环：`F7 / FB / BA / AC`。

调试先后顺序（建议固定流程）

  1. 先自检：固件启动后先看模块加载/初始化是否有报错（先排系统级问题）。
  2. 再驱动：逐项确认关键驱动可工作（ADC、Laser、Optical Switch、APD、PLL）。
  3. 再信号链：先看实时信号是否成立（幅值、内外路语义、切路后变化）。
  4. 然后 AC：确认 APD 搜索链和功率切换链可闭环。
  5. 再 F7：确认身份与当前目标原理一致。
  6. 再 FB/BA：先单次测距，再稳定性测试。
  7. 再 A3：波形与采样语义做交叉确认。
  8. 最后 F1/F4：只在前面都通过后进入校准链。
  9. 每阶段写 runs：只写“已确认/待确认”，通过后再改完成。
